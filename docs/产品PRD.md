# Task Platform (任务编排中心) - 产品需求文档 (PRD)

## 1. 产品概述

### 1.1 背景与目标
在多 Agent 协作和复杂任务流转场景中，需要一个统一的控制面来管理任务分发、状态追踪、异常处理和系统观测。Task Platform 旨在打造一个标准化的**任务编排中心**，它不作为单一的 Agent Runtime，而是作为上层的调度和状态机平台，统一协调各种不同的 Agent（如 OpenClaw、Hermes 等），确保任务能够稳定闭环。

### 1.2 产品定位
- **任务编排中心**：负责任务状态机、流程推进、循环兜底控制、人工干预。
- **协议标准化**：对下对接多种 Agent 平台，统一任务下发与结果回调协议。
- **治理与观测层**：提供全面的运行监控、告警、审计以及资源配额管理。

## 2. 核心模块需求

根据产品规划及现有 UI 原型，系统分为七大核心模块：

### 2.1 数据看板 BI & 运行监控 (Dashboard)
- **核心指标卡片**：
  - 总任务数及环比增长率。
  - 活跃运行中任务数（带实时状态指示）。
  - 任务整体成功率。
  - 队列积压数及状态监控。
- **可视化图表**：
  - **任务执行趋势图**：按小时维度的成功/失败任务数折线图。
  - **Agent 任务负载分布图**：各 Agent (如 OpenClaw, Hermes, Coze 等) 承担的任务比例环形图。

### 2.2 任务中心 (Tasks)
- **任务列表**：
  - 字段展示：任务 ID / 名称、状态（Running, Success, Failed）、分配 Agent、执行进度条。
  - 搜索与筛选：支持按名称搜索，按状态过滤。
- **操作与详情**：
  - 新建任务：定义任务目标、优先级、SLA、标签等。
  - 任务级操作：查看任务详情、查看执行日志（每轮 Agent 响应、工具调用等）、手动重试/终止。

### 2.3 流程编排 (Orchestrator)
- **流程设计器 (DAG)**：
  - 可视化画布：支持拖拽、连线构建任务流。
  - 支持节点类型：
    - 起点节点 (Start Node)：如接收 Webhook、定时触发。
    - Agent 节点 (Agent Task)：将子任务分配给指定 Agent 执行。
    - 终点节点 (End Node)：如输出报告、发送告警。
    - 逻辑节点：支持串行、并行、条件分支、循环。
- **流程管理**：支持“保存草稿”和“发布流程”。

### 2.4 告警中心 (Alerts)
- **告警列表**：展示活跃的系统与任务级告警，包含告警级别 (Critical/Warning)、告警类型、触发内容、触发时间。
- **常见告警规则**：
  - Agent 回调超时 (如超过 300s) / 不可用。
  - 队列积压超过阈值。
  - 任务失败率异常、循环次数异常。
- **操作**：支持对告警进行“确认 / 解决”闭环操作。

### 2.5 治理与审计 (Audit)
- **审计日志列表**：
  - 记录所有关键系统状态变更，便于追责与回放。
  - 字段：时间、操作人/来源（系统自动或用户手动）、动作 (如 `CREATE_TASK`, `AUTO_RETRY`, `UPDATE_AGENT`)、操作对象、详情描述。
- **操作**：支持对操作对象进行检索，支持日志数据导出。

### 2.6 Agent 接入 (Adapters)
- **适配器卡片视图**：
  - 展示接入的 Agent 详情：名称、类型 (如通用智能体、私有化大模型)、运行状态 (Online/Offline)、协议类型 (Webhook Push, Polling Pull, REST API) 和描述。
- **管理功能**：
  - 新增适配器：配置接入协议、鉴权签名、状态映射等。
  - 适配器配置、修改、停用。

### 2.7 MCP 配置 (Model Context Protocol)
- **扩展组件列表**：
  - 展示已接入的 MCP 组件：名称、类型 (如 RAG 知识库、Skill 插件、Memory 记忆服务)、Endpoint URL、连接状态 (Connected, Warning, Error)。
- **管理功能**：
  - 新增 MCP 配置、测试连通性、编辑。

## 3. 业务闭环与核心机制

### 3.1 任务状态机
任务生命周期必须遵循严谨的状态流转：
- `Draft` -> `Ready` -> `Scheduled` -> `Running` -> `WaitingFeedback` -> `Reviewing` -> `Success` / `Failed` / `Cancelled`
- 对于循环型任务增加附加状态：`Iterating` (循环中), `Paused` (暂停), `Terminated` (强制终止)。

### 3.2 循环兜底控制
循环任务必须由平台侧“兜底”，不能完全依赖 Agent 自行判断：
- `max_iterations`: 最大循环次数限制。
- `max_duration`: 最大执行时长限制。
- `no_progress_threshold`: 无进展阈值，防止空转。
- `budget_limit`: 资源/Token 消耗上限。

### 3.3 异常与重试策略
- 任务失败或回调超时后，平台提供标准化动作：**自动重试**、**转人工干预**、**更换 Agent 降级执行**、**彻底终止**。

## 4. 推荐 MVP (Phase 1) 实施范围
1. **前端 WebUI**：完成上述 7 个页面的基本交互（基于 React/Next.js 等）。
2. **核心后端**：实现任务状态机、基础异步调度（基于 Celery/消息队列）、Webhook 回调接收。
3. **Agent 接入**：优先支持 `OpenClaw` (抓取) + `Hermes` (总结) 两个 Agent 适配器。
4. **调度能力**：优先支持 Cron 周期任务与一次性延迟任务。
5. **控制能力**：告警（超时、失败）、循环控制（最大次数、最大时长、人工终止）。
