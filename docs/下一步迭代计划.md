# Task Platform 下一步迭代计划

## 1. 文档目标

本文档用于明确 Task Platform 下一步迭代的方向，并将其整理为可执行的实施计划。

它不替代 PRD，也不重复技术设计文档，而是聚焦以下问题：

- 下一阶段要优先解决什么问题
- 为什么这些问题比其他事项更优先
- 具体要做哪些事情
- 每项事情完成后如何验收
- 实施顺序、依赖关系和风险是什么

本文档与 [关于loop.md](./关于loop.md) 配套使用。前者回答“平台应该往哪里进化”，本文档回答“下一步具体怎么做”。

## 2. 当前判断

结合现有仓库实现和最新设计思路，项目当前已经具备以下基础：

- 基础任务模型、状态机与执行实例
- 定时调度、超时扫描、失败重试
- Agent 适配器接入与回调协议
- DAG 工作流编排
- MCP 配置与探测
- 告警、审计、指标与基本前端页面

这意味着当前阶段的核心矛盾已经不是“能不能跑起来”，而是：

1. 平台还没有显式表达“目标完成标准”
2. 平台还缺少独立的验证与收敛判断机制
3. Loop 的治理信息还没有形成足够清晰的产品闭环

因此，下一阶段不建议优先做更多 Agent 类型，也不建议先做过重的企业能力，而应优先把平台从“任务编排中心”推进为“可验证、可终止、可追踪的 Loop 平台”。

## 3. 迭代总目标

下一阶段的总目标定义为：

> 完成 Task Platform 的第一版 Loop 闭环增强，使平台能够基于明确的完成标准驱动任务持续推进，并在前后端提供可解释的验证、终止与迭代追踪能力。

这个目标拆成三个一级目标：

### 3.1 目标一：补齐 Goal 与 Success Criteria

让任务不仅有自然语言目标，还要有可执行的完成标准。

### 3.2 目标二：补齐 Verifier 与收敛治理

让平台能够判断“是否完成”“是否继续”“是否空转”，而不是只被动接收 Agent 的自我声明。

### 3.3 目标三：补齐可观测的 Loop 体验

让用户在产品界面上看见每轮迭代发生了什么、为什么继续、为什么停止。

## 4. 本轮迭代范围

为控制交付风险，建议把本轮范围限定在以下内容。

### 4.1 必做范围

- 任务模型补充完成标准字段
- 后端新增规则型 Verifier
- LoopGuard 增加基础无进展检测
- Run 详情页展示迭代信息、验证结果和终止原因
- 审计日志补充验证与终止相关事件
- 文档更新，明确 Loop 运行闭环

### 4.2 暂不纳入本轮

- 长期记忆系统
- Skill Registry
- 多租户与 RBAC
- Worktree / Sandbox 隔离执行
- 独立 LLM Verifier Agent
- 成本中心和 Token 预算系统

这些方向重要，但不应打断当前版本最关键的闭环建设。

## 5. 可执行计划

建议把下一步工作拆成 4 个工作包，按顺序推进。

### 5.1 工作包 A：任务目标模型增强

### 目标

让任务定义从“只有 objective”升级为“objective + success criteria”。

### 计划事项

1. 扩展 Task 的数据模型
2. 在创建任务和更新任务接口中支持新的字段
3. 在前端任务创建页和详情页展示完成标准
4. 为历史数据提供兼容策略

### 建议新增字段

- `success_criteria`: JSON 或结构化对象，描述成功条件
- `failure_criteria`: 可选，描述失败判据
- `verification_mode`: 可选，默认 `rule_based`

### 实施说明

- 第一版不追求复杂 DSL，可以先采用结构化 JSON
- 允许用户输入简单规则，例如：
  - `tests_passed = true`
  - `lint_passed = true`
  - `artifact_exists = report.md`
- 对旧任务允许字段为空，为空时沿用当前行为

### 验收标准

- 可以通过 API 创建包含 `success_criteria` 的任务
- 前端创建页能填写并提交完成标准
- 任务详情页可查看完成标准
- 旧任务读取和执行不报错

### 5.2 工作包 B：规则型 Verifier

### 目标

在 Agent 回调之后，不直接以 `status=success` 作为任务成功依据，而是先进入验证阶段。

### 计划事项

1. 新增 Verifier 服务
2. 在 `handle_feedback` 中接入验证流程
3. 新增验证结果记录结构
4. 将验证结论写入审计日志

### 建议实现方式

第一版采用规则验证器，输入为：

- 当前任务配置
- 当前回调状态
- 结构化结果 `result_payload`
- 可选日志摘要

输出为：

- `passed`
- `failed`
- `needs_continue`
- `reason`
- `signals`

### 状态流建议

建议把当前逻辑补齐为：

```text
Agent success -> Reviewing -> Verify
  -> passed -> Success
  -> needs_continue -> Iterating / Scheduled
  -> failed -> Failed 或人工介入
```

### 第一版可支持的规则

- 指定字段存在
- 指定布尔值为真
- 指定状态码命中
- 指定产物存在
- 指定错误字段不存在

### 验收标准

- Agent 回调 `success` 时，不再直接必然进入 `Success`
- 验证失败时能记录失败原因
- 验证要求继续时，系统能进入下一轮
- 每次验证都能在审计中找到记录

### 5.3 工作包 C：LoopGuard 与收敛检测增强

### 目标

让平台具备最基础的“防空转”能力。

### 计划事项

1. 在当前 `max_iterations`、`max_duration` 之外新增 `no_progress_threshold`
2. 为每轮执行保存简化进展快照
3. 检测连续多轮无明显变化时自动终止或告警

### 建议实现方式

第一版不做复杂语义分析，可以先做轻量规则：

- 比较连续两轮 `result_payload` 是否完全一致
- 比较关键字段是否发生变化
- 比较日志摘要是否重复

当连续 N 轮无明显变化时：

- 记录审计事件
- 触发告警
- 将任务标记为失败或终止

### 验收标准

- 可以为任务配置 `no_progress_threshold`
- 连续无进展达到阈值时能自动被识别
- 前端可看到任务是因为“无进展终止”
- 告警中心能看到对应告警

### 5.4 工作包 D：前端 Loop 可视化补齐

### 目标

让用户能直接看懂一次 Loop 是如何推进和停止的。

### 计划事项

1. 在任务详情或运行详情页增加“迭代记录”区域
2. 展示每轮的输入摘要、输出摘要、验证结果、状态变化
3. 展示终止原因
4. 展示当前完成标准

### 建议展示内容

- 当前目标
- 完成标准
- 已执行轮次 / 最大轮次
- 最近一次验证结果
- 最近一次告警
- 终止原因
- 事件时间线

### 验收标准

- 用户能在页面看到每轮执行和验证记录
- 用户能看到任务为何停止
- 用户能区分“Agent 成功”和“平台判定完成”两个概念

## 6. 推荐实施顺序

建议按以下顺序执行：

1. 工作包 A：任务目标模型增强
2. 工作包 B：规则型 Verifier
3. 工作包 C：LoopGuard 与收敛检测增强
4. 工作包 D：前端 Loop 可视化补齐

原因如下：

- 没有完成标准，就无法做可靠验证
- 没有验证层，Loop 只是“重复执行”
- 没有收敛治理，Loop 容易失控
- 没有前端呈现，产品价值无法被用户感知

## 7. 迭代排期建议

如果按一个小版本推进，建议拆成 3 个里程碑。

### M1：模型与接口打底

建议周期：3 到 5 天

交付内容：

- Task 模型补字段
- Schema 与 API 更新
- 创建 / 编辑页支持完成标准
- 基础兼容逻辑

### M2：验证与收敛闭环

建议周期：5 到 7 天

交付内容：

- 规则型 Verifier
- `handle_feedback` 接入验证链路
- `no_progress_threshold` 支持
- 审计与告警扩展

### M3：前端可视化与联调收尾

建议周期：3 到 5 天

交付内容：

- 运行详情页迭代视图
- 终止原因展示
- 联调、自测、文档更新

## 9. 风险与依赖

### 9.1 主要风险

- `success_criteria` 设计过重，导致实现周期拉长
- 状态机改造不当，导致现有任务流转回归
- 前后端字段定义不一致，影响联调
- 无进展检测过于粗糙，误判正常任务

### 9.2 控制策略

- 第一版只做规则型、轻量结构
- 保持对旧任务的兼容路径
- 先补自动化测试，再推进状态流改造
- 将无进展检测设计成可配置、可关闭

### 9.3 关键依赖

- 数据模型和迁移方案先定稿
- 前后端完成字段契约对齐
- 需要至少一个可稳定复现的 Adapter 回调链路用于联调

## 10. 本轮完成定义

当以下条件全部满足时，可以认为本轮迭代完成：

1. 用户能创建带完成标准的任务
2. 平台能在回调后执行规则验证
3. 平台能识别基础无进展状态
4. 任务详情页能展示迭代与终止原因
5. 关键链路具备对应测试
6. 相关设计文档和使用文档已更新

## 11. 下一轮展望

本轮完成后，再进入下一阶段会更稳妥，优先考虑：

- 独立 Verifier Agent
- 工作流级人工审批节点
- 长期记忆抽象
- Skill / Playbook 资产化
- 成本预算与策略路由

在那之前，当前版本的首要任务不是“变得更复杂”，而是“把闭环真正跑稳”。
