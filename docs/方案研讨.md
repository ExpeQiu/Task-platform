
**产品蓝图**
- **任务中心**: 创建任务、设置优先级/SLA、标签、负责人、关联项目、执行策略。
- **流程编排**: 支持串行、并行、条件分支、循环、人工审批节点、回滚节点。
- **Agent 接入层**: 对接多种 Agent 平台，统一成标准任务协议与回调协议。
- **调度中心**: 定时任务、事件触发、重试策略、并发控制、资源配额。
- **执行监控**: 查看任务状态、每轮执行日志、Agent 响应、工具调用、失败原因。
- **治理中心**: 审计、权限、告警、运行指标、成本统计、运维控制台。

**推荐总体架构**
- **前端 WebUI**: 任务列表、任务详情、流程设计器、运行监控、告警中心、审计查询。
- **API 网关**: 对外提供任务管理、调度管理、回调接收、认证鉴权。
- **任务编排服务**: 负责任务状态机、流程推进、循环控制、人工干预。
- **调度服务**: 处理 cron、延迟任务、周期任务、超时任务、重试任务。
- **Agent 适配层**: 每类 Agent 平台一个 adapter，屏蔽协议差异。
- **事件总线**: 用于任务状态变化、告警事件、回调事件分发。
- **存储层**: MySQL/Postgres 存元数据，Redis 做队列/缓存，对象存储放日志与产物。
- **观测层**: 日志、指标、链路追踪、审计记录、告警规则。

**任务闭环设计**
- **任务建模**
  - Task: 业务任务主体
  - TaskRun: 某次实际执行
  - Step: 任务中的子步骤/节点
  - Assignment: 分配给哪个 Agent/执行器
  - Feedback: Agent 回传结果、状态、产物、错误
- **状态机建议**
  - Draft -> Ready -> Scheduled -> Running -> WaitingFeedback -> Reviewing -> Success / Failed / Cancelled
  - 对循环型任务增加 Iterating、Paused、Terminated
- **闭环关键点**
  - 创建时定义完成标准，而不是只给自然语言目标
  - 执行中保留每一步输入、输出、决策依据
  - 失败后有重试、转人工、换 Agent、终止四种标准动作
  - 结果归档后可形成知识沉淀与复盘数据

**Agent 对接方案**
- **统一协议**
  - 平台发出: 任务 ID、上下文、目标、约束、超时、回调地址
  - Agent 回传: 当前状态、阶段结果、日志、产物、错误码、建议下一步
- **接入模式**
  - Pull 模式: Agent 主动拉取任务
  - Push 模式: 平台推送任务给 Agent
  - Hybrid 模式: 平台编排、Agent 拉取执行
- **适配器职责**
  - 协议转换
  - 鉴权签名
  - 超时与重试
  - 状态映射
  - 日志标准化
- **建议**
  - 第一版先统一成“异步任务 + 回调”模型，最稳妥
  - 不要把平台直接耦合到某一类 Agent Loop 实现

**定时任务与循环控制**
- **定时任务**
  - 支持 cron、一次性延时、固定间隔、事件触发
  - 支持幂等键，避免重复调度
  - 支持错过触发补偿策略
- **循环控制**
  - max_iterations: 最大循环次数
  - max_duration: 最大执行时长
  - no_progress_threshold: 无进展阈值
  - budget_limit: Token/成本上限
  - manual_interrupt: 人工暂停/终止
- **推荐做法**
  - 把循环视为“受控状态机”，不要只靠 Agent 自己判断是否结束
  - 平台层必须有防空转、防爆预算、防重复动作机制

**审计与运维**
- **审计**
  - 谁创建/修改/终止任务
  - 谁审批过流程
  - 哪个 Agent 在何时执行了什么
  - 输入输出、关键参数、状态变更留痕
- **运维**
  - 队列积压监控
  - 回调失败监控
  - Agent 在线状态与健康检查
  - 任务超时、重复重试、异常终止统计
- **建议**
  - 所有状态变更采用 event log 记录，便于追责与回放

**告警机制**
- **告警类型**
  - 任务失败率升高
  - 某 Agent 回调超时/不可用
  - 循环次数异常
  - 成本异常上升
  - SLA 超时
  - 队列积压
- **告警通道**
  - 站内通知
  - 邮件
  - 企业微信/钉钉/Slack/Webhook
- **告警策略**
  - 阈值告警
  - 趋势告警
  - 组合告警
  - 抑制与去重
  - 升级机制

**推荐 MVP**
- WebUI：任务创建、列表、详情、运行日志、手动重试/终止
- 核心后端：任务状态机、异步调度、回调接收、基础审计
- Agent 接入：先做 openclaw + hermes-agent 两个 adapter
- 定时能力：先支持 cron + 一次性延迟
- 告警能力：先支持任务失败、超时、回调失败
- 循环控制：先支持最大次数、最大时长、人工停止

**技术选型建议**
- 前端: React + Ant Design Pro / Next.js
- 后端: FastAPI 或 NestJS
- 调度: Celery/Arq/RQ 或基于消息队列实现
- 队列: Redis Streams / RabbitMQ / Kafka
- 数据库: PostgreSQL
- 观测: Prometheus + Grafana + Loki/ELK
- 工作流增强: 后续可引入 Temporal / LangGraph 风格状态编排

**分阶段实施**
- **Phase 1**: 任务中心 + 基础调度 + Agent 回调 + 日志审计
- **Phase 2**: 流程编排 + 循环控制 + 告警中心 + 人工审批
- **Phase 3**: 多 Agent 协作、策略路由、成本优化、知识沉淀
- **Phase 4**: 多租户、权限体系、SLA、运营报表、自动扩缩容

**几个关键决策**
- 平台是“任务编排中心”，不是“单一 Agent Runtime”
- 平台层统一状态机，Agent 只负责执行与反馈
- 循环必须由平台兜底控制
- 审计、告警、回调可靠性要从第一版就纳入设计
