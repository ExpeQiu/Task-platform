# Task Platform (任务编排中心) - 技术设计文档

## 1. 系统总体架构

根据前期方案研讨，系统采用分层架构设计，确保核心编排逻辑与底层 Agent 执行引擎的解耦：

### 1.1 架构分层
1. **前端展示层 (WebUI)**
   - 提供可视化操作界面，包括 Dashboard、任务管理、DAG 流程设计器、告警审计与配置中心。
2. **接入与网关层 (API Gateway)**
   - 统一入口，提供 HTTP/REST API 与 Webhook 接收点。
   - 负责统一认证鉴权、限流策略。
3. **核心业务层 (Core Services)**
   - **任务编排服务**：维护任务状态机、推进流程流转、处理循环控制、分支路由。
   - **调度服务**：基于时间（Cron、延迟）或事件触发任务投递，处理超时监控与自动重试补偿。
4. **适配与扩展层 (Adapter & MCP Layer)**
   - **Agent 适配器 (Adapters)**：针对 OpenClaw、Hermes、Coze 等不同 Agent 平台实现独立的 Adapter，将异构协议转化为平台标准协议。支持 Push (Webhook) 与 Pull (Polling) 模式。
   - **MCP 扩展**：对接外部能力（如 RAG 知识库、Skill、Mem0），通过统一接口为 Agent 提供工具调用支持。
5. **事件总线 (Event Bus)**
   - 异步解耦任务状态变化、回调事件分发和告警触发。
6. **存储与观测层 (Storage & Observability)**
   - 关系型数据：存储任务元数据、流转记录、审计日志。
   - 缓存/队列：暂存高并发任务队列，维护分布式锁和速率限制。
   - 观测：业务指标（Prometheus）、全链路日志追踪。

## 2. 技术选型建议

- **前端框架**: React + Ant Design Pro / Next.js (当前 Demo 使用 TailwindCSS 构建原型)。
- **后端服务**: FastAPI (Python) 或 NestJS (Node.js)。
- **任务调度与队列**: Celery / Arq 或基于 Redis Streams / RabbitMQ 构建。
- **持久化存储**: PostgreSQL (首选) 或 MySQL。
- **缓存层**: Redis。
- **日志与观测**: Prometheus + Grafana + Loki (或 ELK)。
- **高级工作流引擎 (备选)**: 后期复杂编排可引入 Temporal 或 LangGraph 风格的底层状态引擎。

---

## 3. 核心领域模型 (Domain Model)

- **Task (任务模板)**: 业务维度的定义，包含任务目标、默认配置、SLA、最大循环次数、超时配置等。
- **TaskRun (执行实例)**: Task 的一次具体执行记录，持有完整的状态机生命周期和上下文变量。
- **Step / Node (执行节点)**: 任务流程中的具体子步骤，区分系统节点（Start/End/条件判断）与 Agent 节点。
- **Assignment (分配记录)**: 记录某个 Step 被分派给哪个具体的 Agent Adapter 执行。
- **Feedback (回调反馈)**: Agent 执行后回传的结果，包括当前状态、产物、错误码、日志。

## 4. 关键技术方案设计

### 4.1 统一 Agent 对接协议与回调模型
系统第一版采用**“异步任务 + 回调”**的标准模型，不强耦合任何一个 Agent 的内部 Loop 实现。
- **平台下发请求**：
  ```json
  {
    "task_id": "TSK-1001",
    "run_id": "RUN-54321",
    "objective": "抓取并分析竞品数据",
    "context": { ... },
    "constraints": { "timeout": 300, "max_tokens": 8000 },
    "callback_url": "https://api.task-platform.internal/v1/webhooks/agent_feedback"
  }
  ```
- **Agent 异步回调**：
  ```json
  {
    "run_id": "RUN-54321",
    "status": "success", // success, failed, requires_action
    "result_payload": { ... },
    "logs": [ ... ],
    "error_code": null
  }
  ```

### 4.2 任务状态机设计
通过严谨的状态流转防止脏数据和并发冲突：
1. `Draft` (草稿) -> `Ready` (就绪) -> `Scheduled` (已调度入队)。
2. Agent 开始执行后转为 `Running`，若等待人类输入或子任务，转入 `WaitingFeedback`。
3. 执行完成经过结果校验（`Reviewing`）后，流转到终态：`Success`、`Failed` 或 `Cancelled`。
- **实现机制**：所有状态流转操作必须带有版本号（Optimistic Locking）或在数据库层加锁，流转成功后触发 Event Log 写入审计系统。

### 4.3 循环兜底与防爆防空转机制
对于自我驱动的 Agent 循环任务，平台层必须作为最高仲裁者：
- **拦截器机制**：在每次调度下发新一轮迭代前，平台会校验 `TaskRun` 上的：
  1. `iteration_count` 是否达到 `max_iterations`。
  2. 距离开始时间是否超出 `max_duration`。
  3. 通过 LLM 验证或文本 Diff 判断是否命中 `no_progress_threshold`（陷入死循环）。
- **处理方式**：一旦越界，平台强制掐断任务，状态置为 `Failed`，并抛出 `Critical` 级别告警。

### 4.4 定时调度与告警触发
- **定时与重试**：基于 Redis / Celery Beat 轮询任务队列。对于执行失败的任务，根据其配置的重试策略（如指数退避）重新投递。
- **超时告警扫描**：独立的 Watchdog 服务定期扫描处于 `Running` 状态的记录。若当前时间减去 `last_updated_time` 大于超时阈值，则生成一条 Alert 记录，通过 WebSocket 推送至前端（UI 告警红点），并通过钉钉/邮件通道分发。

## 5. 分阶段实施路径 (Phases)

- **Phase 1 (MVP 阶段)**：完成核心任务状态机、基础调度、WebUI 展示、日志审计，跑通 OpenClaw 和 Hermes 的异步回调链路。
- **Phase 2 (进阶编排)**：上线可视化流程编排 (DAG)，实现循环控制、高级告警中心、人工审批节点。
- **Phase 3 (生态与协作)**：引入多 Agent 协作、MCP 工具路由，实施成本优化策略与知识沉淀体系。
- **Phase 4 (企业级能力)**：多租户隔离、RBAC 权限体系、自动化 SLA 运营报表及节点自动扩缩容。

## 7. 本地运行

```bash
cp .env.example .env
./scripts/start.sh
./scripts/verify.sh
```

Mock Adapter 默认指向 `packages/agent-simulator`（Push + Pull）。切换真实 Agent 时仅需替换 Adapter `endpoint` 并实现对应协议。
